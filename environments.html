Gym environments

Upkie has reinforcement learning environments following the Gymnasium API. All of them are single-threaded and run as-is in both simulation and on real robots. Each environment has its own observation and action spaces, but all of them provide access to full spine observation via info dictionaries. See Observations for details.

Ground velocity

In the UpkieGroundVelocity environment, Upkie keeps its legs straight and actions only affect wheel velocities. This environment is used for instance by the MPC balancer and PPO balancer agents. Its action consists of a 1D vector containing the commanded ground velocity, which is then internally converted to wheel velocity commands.

\[ \begin{align*} o &= \begin{bmatrix} \theta \\ p \\ \dot{\theta} \\ \dot{p} \end{bmatrix} & a &= \begin{bmatrix} \dot{p}^* \end{bmatrix} \end{align*} \]

where we denote by:

  • \(\theta\) the pitch angle of the base with respect to the world vertical, in radians.
  • \(p\) the position of the average wheel contact point, in meters.

Check out the UpkieGroundVelocity documentation for more details.

Servos

The UpkieServos environment has dictionary observation and action spaces. Observation dictionaries return position, velocity and torque for each servo. Action dictionaries specify target positions, target velocities and feedforward torque commands that are directly sent to the moteus controllers. Each motor controller will then apply a standard control law with feedforward torque and position-velocity feedback:

\[ \begin{align*} \tau & = \tau_{\mathit{ff}} + k_{p} k_{p}^{\mathit{scale}} (\theta^* - \theta) + k_{d} k_{d}^{\mathit{scale}} (\dot{\theta}^* - \dot{\theta}) \end{align*} \]

where for joint \(i\) we denote by:

  • \(\tau\) is the commanded joint torque,
  • \(\tau_{\mathit{ff}}\) the feedforward torque (set only this one for direct torque control),
  • \(\theta\) the measured joint position,
  • \(\dot{\theta}\) the measured joint velocity,
  • \(\theta^*\) the target joint position,
  • \(\dot{\theta}^*\) the target joint velocity.

Position and velocity gains \(k_{p}\) and \(k_{d}\) are configured in each moteus controller directly; we can update the overall feedback gains via the normalized parameters \(k_{p}^{\mathit{scale}} \in [0, 1]\) and \(k_{d}^{\mathit{scale}} \in [0, 1]\).

Check out the UpkieServos documentation for more details.

Servo positions

The UpkieServoPositions has the same observation space as UpkieServos, but the action space is only composed of \(\theta\), \(k_d\) and \(k_p\). This makes the agent command servo positions with velocity damping:

\[ \begin{align*} \tau & = k_{p} k_{p}^{\mathit{scale}} (\theta^* - \theta) - k_{d} k_{d}^{\mathit{scale}} \dot{\theta} \end{align*} \]

Servo torques

The UpkieServoTorques has the same observation space as UpkieServos, but the action space is only composed of \(\tau_{\mathit{ff}}\). This makes the agent command joint torques directly:

\[ \begin{align*} \tau & = \tau_{\mathit{ff}} \end{align*} \]
